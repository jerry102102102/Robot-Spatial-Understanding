# Understand Robot Spatial

Ground every spatial claim in deterministic source parsing and forward kinematics. Treat generated JSON as bound machine artifacts, concept clauses as proof-carrying symbolic abstractions with explicit modality, and generated Markdown as compact reasoning context.

## Fast path: judge one execution result

Use this path when the user asks whether a simulated or ROS action ran, succeeded, failed, or produced the intended result.

1. Prefer an existing immutable capture or simulation output. Do not run `execute-capture` merely to answer a question; it can dispatch a goal. Use it only when the user explicitly requests dispatch and supplies the exact configured action-instance authorization token.
2. Require the exact `functional-model.json`, action adapter config, capture, and any separately typed condition/effect evidence. Reject files whose model, config, capture, goal, source, clock, action-instance, or artifact digests do not bind.
3. Run `ros_action_adapter.py normalize` for a ROS action capture. Inspect the normalization report for ignored goals, unknown or duplicate statuses, conflicting publishers, missing publisher identity, client/server time separation, and feedback/result non-promotion.
4. Run `robot_spatial.py action-assurance`, then `verify-action-assurance`. Treat a failed verification as corrupted or mismatched evidence, not as an action failure.
5. Create a `robot-spatial-action-assurance-query.v1` with `intent: summarize_action`, run `query-action-assurance`, and use `explain_lifecycle`, `explain_precondition`, `explain_effect`, or `why_not_ready` only when the summary needs detail. Read [references/action-assurance-contract.md](references/action-assurance-contract.md) and [references/ros-action-adapter-contract.md](references/ros-action-adapter-contract.md) before interpreting the result.
6. Report the verdict in the user's language with these separate fields:
   - bound action, actor, target, decision time, and evaluation time;
   - evidence integrity and provenance;
   - precondition readiness at decision time;
   - goal response, latest status, terminal result, and lifecycle consistency;
   - each declared effect observation and whether it occurred after execution began;
   - discrepancies, missing evidence, stale evidence, and authority/clock limitations;
   - bounded conclusion: protocol outcome, observed effect outcome, and what remains unestablished.
7. Never collapse these claims:
   - `ready_under_declared_model_and_evidence` is not dispatch authorization or safety;
   - accepted/executing is not completion;
   - `result_succeeded` is an action-server report, not physical success;
   - a later effect observation does not prove the action caused it;
   - feedback or result payload is audit data, not effect evidence;
   - missing evidence means unknown, not false, unless a declared complete closed-world projection permits a negative.

For simulation, call the result established only within the declared simulator, model, clock, sensors, evidence sources, and sampled interval. Do not upgrade simulation evidence to real-hardware truth.

## Workflow

1. Resolve the authoring source. For a ROS source workspace or any project with `package.xml`, run `prepare` with the source entrypoint, workspace root, Xacro executable, and explicit variant mappings; then read `prepare.json` before the generated context. It discovers unambiguous source packages, logs Xacro package lookups, hashes the conservative used-package source closure, validates `resolved.urdf`, and exports one context. Read [references/project-preparation-contract.md](references/project-preparation-contract.md) before interpreting its provenance or runtime boundary. Use `expand-xacro` directly only when package resolution is already established. Never parse unexpanded Xacro as URDF.
2. Locate this skill's `scripts/robot_spatial.py` and run `validate` with URDF, SRDF, and semantic annotations that exist before reasoning about the model.
3. Create pose input as JSON when the question depends on joint values. Use radians and meters:

   ```json
   {"pose_name": "home", "joints": {"shoulder": 0.0, "slide": 0.15}}
   ```

4. If project files do not explicitly identify roles such as base, planning frame, flange, TCP, manipulator group, or end effector, create a `semantics.json` using the contract reference. Confirm role choices with the user when they materially affect the task; do not derive them from names alone.
5. Create a package map when meshes use `package://` URIs. Map each ROS package name to its package root directory.
6. When the question depends on physical-world placement, create a separate `robot-spatial-world-scene.v1` snapshot. Bind exactly one URDF robot instance by `robot_name` and `root_link`; declare its root mount, scene frames, optional gravity, external objects, collision geometry, snapshot ID, and provenance. Use typed `scene_frame/`, `scene_object/`, `scene_geometry/`, `robot_frame/`, and `robot_geometry/` identities. Never relabel robot-local `frame/X` as physical world state. Read [references/world-scene-contract.md](references/world-scene-contract.md) before interpreting world transforms, scene gravity loads, or robot/environment collision.
7. When the question asks what was observed at time `t`, add a digest-bound `robot-spatial-observation-log.v1` or v2 plus `robot-spatial-observation-query.v1`. Bind the exact URDF semantic digest and scene digest; normalize joint, root, and declared-object pose streams; set per-stream maximum ages, required objects, and explicit root/object fallback policy. The resolver selects only the latest sample at or before `t` with zero-order hold. For ROS 2 `/joint_states`, `/tf`, or `/tf_static`, first run `ros_observation_adapter.py`: create a model/scene-bound config, preserve an immutable capture, then normalize it into v2. Reject clock/config mismatch, future header stamps, conflicting joint values, multiple visible authorities, TF parent switches/cycles, static/dynamic child reuse, stale TF paths, and incomplete/stale joint assemblies instead of arbitrating silently. Keep model declarations, static-scene declarations, transport capture, and normalized observations distinct. Read [references/ros-observation-adapter-contract.md](references/ros-observation-adapter-contract.md) and [references/temporal-observation-contract.md](references/temporal-observation-contract.md) before calling ROS data current or interpreting observation-conditioned transforms, gravity, or collision.
8. Run `export --inspect-meshes --render --motion-atlas` into a task-local artifact directory when every visual/collision shape and joint-as-cause relation matters, or pass the same inspection options to `prepare`. Every URDF export also writes pose-independent `articulation-grammar.json`, proof-carrying `concept-graph.json`, and controlled `concept-language.rsl`; compile supported SDF/MJCF directly with `articulation-grammar`. When the source tree is only a coordinate parameterization of a closed-chain, parallel, or coupled mechanism, first compile the exact grammar, author a digest-bound `robot-spatial-constraint-spec.v1`, then run `constraint-graph`, `evaluate-constraints`, and `verify-constraint-graph`; pass the spec to `export` or `prepare` so the concept graph includes asserted relation clauses and exact driver-dependency derivations. When one local pose is insufficient, author a graph-digest-bound `robot-spatial-configuration-atlas-spec.v1`, generate and verify a finite atlas, and pass both specs to export/prepare. When the question asks what a component is for or what action it affords, first inspect exact concept IDs, then author a digest-bound `robot-spatial-function-affordance-spec.v1` and pass `--functional-spec` to export/prepare. This writes `functional-model.json`; use `query-functions` for a task-relevant functional plus recursive structural proof closure. When the question concerns one attempted action's readiness, goal/status/result, or declared effects, author a digest-bound `robot-spatial-action-evidence-bundle.v1`, compile `action-assurance`, verify it, and use `query-action-assurance`. For a real ROS 2 action exchange, first use `ros_action_adapter.py`: bind one exact functional action to a canonical action name/type, preserve the goal UUID/payload and client exchange, normalize service/topic reports offline, and add separately typed condition/effect sources. Never infer goal acceptance from status-only traffic, promote feedback/result payload to effect evidence, or treat `SUCCEEDED` as physical success. `execute-capture` can move hardware and requires `--authorize-dispatch` equal to the exact action instance; this is only an explicit dispatch gate, not a safety or organizational authorization proof. Select conditions at decision time and lifecycle/effects at evaluation time; never equate readiness with dispatch authorization, result success with physical success, or post-execution observation with causation. Read `agent-context.json` first; use `query-concepts` for structure, `query-functions` for explicit function knowledge, and action assurance only for the bound action instance, then load only relevant typed cards/facts. Never infer function from names/topology/geometry or promote structural grounding to physical execution. Never promote the spanning tree, a semantic assertion, a functional declaration, an action-server report, or a finite proximity graph to physical/global truth. For cross-format equivalence, use the typed correspondence workflow. For visual/finite cause grounding, use the render/motion atlases. Read [references/concept-language-contract.md](references/concept-language-contract.md), [references/function-affordance-contract.md](references/function-affordance-contract.md), [references/action-assurance-contract.md](references/action-assurance-contract.md), [references/ros-action-adapter-contract.md](references/ros-action-adapter-contract.md), [references/articulation-grammar-contract.md](references/articulation-grammar-contract.md), [references/constraint-graph-contract.md](references/constraint-graph-contract.md), [references/configuration-atlas-contract.md](references/configuration-atlas-contract.md), [references/cross-representation-contract.md](references/cross-representation-contract.md), [references/render-atlas-contract.md](references/render-atlas-contract.md), [references/motion-atlas-contract.md](references/motion-atlas-contract.md), and [references/agent-context-contract.md](references/agent-context-contract.md) before changing this load order.
9. For a natural-language question, first classify it as a symbolic structural composition, project-declared function/capability/relational affordance, time-qualified action readiness/lifecycle/effect evidence, pose-independent tree law, asserted mechanism constraint, constraint evaluation/local solution, finite configuration branch/rank witness, cross-representation comparison, one FK evaluation, an infinitesimal derivative, or a finite endpoint; then normalize typed entities, source/import contract, pose/query/decision/evaluation time, frame direction, snapshot, clock/ages, signed limit status, sampling/seeds/scales/thresholds, projection convention, and answerability. For structure, map the request to a strict `robot-spatial-concept-query.v1` and run `query-concepts`; preserve every supporting clause's modality and use an exact negative only inside a declared complete closed-world projection. For function, map the request to a strict `robot-spatial-functional-query.v1` and run `query-functions`; distinguish project assertion, requirement status/closure basis, unevaluated condition, intended effect, inventory scope, and physical executability. For a bound action instance, map the request to `robot-spatial-action-assurance-query.v1` and run `query-action-assurance`; preserve condition selection reason, protocol state, effect timing, discrepancies, producer/provenance boundary, causal unknowns, and lack of dispatch authorization. Use the remaining exact commands for numeric or newly bound results. State whether a constraint is asserted, whether a residual is satisfied at the named binding, whether mobility is pose-conditioned numerical evidence, and whether any component/rank-drop label is only relative to finite declared sampling. Answer in the user's language and state every binding, frame, unit, digest, coverage, and unresolved boundary that materially conditions the answer. Read [references/grounded-question-contract.md](references/grounded-question-contract.md) before ambiguous or benchmarked questions.
10. Before editing, load or create a project-owned `robot-spatial-invariants.v1` contract for relationships that must remain true. Bind the exact world scene when assertions depend on mounting, world gravity, or external collision; bind the exact log/query pair for temporal readiness, effective transform, or nominal observed-collision assertions. Run `check-invariants` before and after the edit; treat any non-zero exit as rejection. Read [references/invariant-contract.md](references/invariant-contract.md) before authoring or changing these assertions.
11. To measure read-only understanding, add `--generate-evaluation --evaluation-key-out <evaluator-private>/answer-key.jsonl`. Put that private path outside every filesystem/context surface available to the candidate. Provide only the sources, skill, public manifest, questions, and answer template; then grade candidate JSONL. For a cross-family claim, build and grade an independently keyed multi-robot suite. Use `generated_context` to test representation comprehension and `raw_sources` to test autonomous source-to-representation understanding. A raw task may declare confined world/observation paths, a function/affordance spec, action-assurance bundles or ROS action captures/configs bound to the exact exported functional model, an articulation bundle with typed comparisons, `constraint_graphs` naming articulation sources plus digest-bound supplemental specs, and `configuration_atlases` naming those graphs plus raw digest-bound atlas specs; never publish generated grammars, concept graphs/languages, lifecycle evidence sources, action bundles, functional/action-assurance models, normalization reports, constraint graphs, atlases, evaluations, solutions, or comparison reports. Functional evaluation must test explicit component purpose, capability grounding versus physical truth, relational preconditions/effects, and complete/incomplete inventory boundaries. Action-evidence evaluation must separately test decision-time readiness, goal/status/result lifecycle, feedback/result non-promotion, effect timing, discrepancies, content provenance, clock/authority visibility, causation, physical truth, authorization, and safety boundaries. For a source workspace, declare `workflow: prepare` and `workspace_roots`; the candidate must create a new prepared root. Export fresh context under the submission root and never use `--generate-evaluation` as the candidate. Read [references/evaluation-suite-contract.md](references/evaluation-suite-contract.md) before constructing either mode or interpreting a score.
12. To measure edit competence, create a public `robot-spatial-edit-task.v1` and isolated private `robot-spatial-edit-key.v1`, then grade the candidate's edited URDF and invariant contract with `scripts/spatial_edit_evaluation.py`. Include every package map, mesh, scene, observation artifact, or other immutable dependency in `inputs.protected_files`. For supported topology edits, require a `robot-spatial-graph-change-set.v1`, compile it with `scripts/spatial_graph_edit.py`, and submit both the change set and compiled URDF; the grader must reproduce the complete candidate tree from the pinned baseline. Use `scripts/spatial_edit_suite.py` to aggregate multiple independently keyed edit categories. A pass requires requested spatial outcomes, exact source and design-intent allowlists, protected-file integrity, and all updated/protected invariants. Read [references/edit-evaluation-contract.md](references/edit-evaluation-contract.md) and [references/graph-change-contract.md](references/graph-change-contract.md) before constructing a task or interpreting a score.
13. For high-assurance, unfamiliar robot families, action execution evidence, closed/coupled mechanisms, finite branch/singularity claims, or cross-format claims, run the independent concept-graph, function-model, action-assurance, ROS-action-normalization, articulation, constraint-graph, configuration-atlas, and cross-representation source-generator/oracles plus the optional yourdfpy FK, mass-property, gravity-load, trimesh geometry, FCL collision, world-scene, temporal-selection, semantic-render, and counterfactual-motion cross-checks. Preserve reports, versions, digests, mappings, probes/poses, goal UUID/payload hashes, publisher-identity visibility, decision/evaluation times, evidence policies, chart samples/seeds/scales/thresholds, negative controls, solver selections, mimic-limit/joint-anchor cases, tolerances, coverage, and exclusions.
14. After editing URDF, a scene, log normalization, or query policy, rerun `validate`, `export --invariants`, and every query used to justify the change. Run `compare` on before/after `model.json` artifacts and explain intended and unintended robot-local, scene-mount, time-selection, fallback, world-gravity, and collision changes. Update an invariant only when approved design intent changed, never merely to make a regression pass.

## Reasoning Rules

- Distinguish the kinematic structure from mesh shape. A mesh URI and scale do not reveal its actual bounds or surface.
- Make mesh-shape claims only from records marked `measured`. The portable engine measures STL and OBJ; stop on unsupported formats instead of guessing. A collision-only pass does not measure visual meshes: check `requested_kinds`, `by_kind`, and `complete_for_all_declared_meshes` before generalizing.
- Distinguish link, pre-motion joint, visual, collision, and inertial frames. They may share numbers at zero pose but have different semantics.
- Treat `inertial/<link>` as the URDF-declared center-of-mass and inertia-coordinate frame, not the link origin and not a geometry-derived centroid. Use `mass-properties` to combine tensors with FK, tensor rotation, and the parallel-axis theorem; never average link origins. If any selected declaration is invalid or incomplete, preserve `indeterminate` and null aggregate values instead of summing the remaining links.
- Call the result “declared mass properties,” not the robot's actual mass. Report `missing_inertial_links`; an absent inertial element does not prove zero physical mass, and URDF cannot establish payload, calibration-to-hardware, or physical-world completeness.
- Use `gravity-loads` only for pose-conditioned gravity-only statics. State the gravity vector and frame. `generalized_gravity_force` is exerted along positive independent-joint motion; `ideal_static_holding_effort` is its opposite. Missing inertials are not zero load; invalid/incomplete selected inertials make the result `indeterminate`. Do not add damping, friction, acceleration, contacts, payload, transmission loss, motor capability, controller behavior, or actual mounting orientation to this claim.
- Use `scene-gravity-loads` when mounting orientation and world-expressed gravity are explicitly declared. Keep the scene digest and snapshot ID with the result. Do not substitute this for physical calibration or current-world evidence.
- Treat `ros2_control`, legacy `<transmission>`, hardware plugin, command/state interface, mechanical reduction, and joint `<dynamics>` records as expanded-URDF declarations only. Preserve nonstandard dynamics attributes as uninterpreted strings. Neither a declaration nor a modeled-load-versus-effort-limit comparison proves plugin installation, controller configuration, interface claiming, connected hardware, calibration, or command execution.
- Use typed entity IDs in the agent context pack. `link/X` and `frame/X` are deliberately distinct; reject an ambiguous bare name instead of choosing one.
- Keep robot-local and snapshot-bound namespaces distinct. `frame/X` uses the URDF root as a mathematical origin; `robot_frame/X` is the mounted frame in one declared scene. Omitted scene objects are unknown, not evidence of empty physical space.
- Keep model, static-scene, and observation layers distinct. A selected joint/root/object sample is an observation report; an effective pose sourced from `static_scene_declaration` is a fallback hypothesis, never an observed pose.
- For state at time `t`, select only the latest sample whose timestamp is at or before `t`. Preserve clock domain, query time, sample ID, age, maximum age, future-sample ignored count, required objects, and fallback policy. Never use a future sample to answer an earlier query or call a stale/missing stream current.
- For ROS-normalized state, additionally preserve capture/config digests, header-versus-receipt timestamp origin, joint-component ages, TF path edges/ages, publisher-identity visibility, and conflict rejection. An unrelated TF update must never refresh a root/object pose, and a topic/GID label must never be promoted to source truth.
- Treat `current` as an age-policy result only. It does not validate the sensor, clock synchronization, calibration, covariance, data association, object-set completeness, or physical truth.
- Call observation-conditioned geometry results nominal. Even when every required sample is current and every declared pair is collision-free, keep physical collision and safety `not_established`; version 1 checks no uncertainty envelope, omitted object, or swept interval between samples.
- Treat `summary_cnl` as a retrieval index. Cite its bound provenance facts or a fresh deterministic query; never cite compact wording as an independent oracle.
- Treat the concept graph as the compositional symbolic layer. Use its typed clauses and proof closure for roots, unique paths, descendants, branch points, structural leaves, maximal serial segments, driver effects, frame laws, constraint dependencies, and finite-node comparison. A clause's modality is part of its meaning: never rewrite `project_asserted` or `supplemental_asserted_relation` as derived physical fact.
- Return a negative concept answer only inside an explicitly complete closed-world projection: validated tree membership/topology or complete supported articulation driver/operator/frame dependencies. Missing semantic roles, functions, mechanism relations, physical state, global branches, or safety evidence are unknown rather than false. Read [references/concept-language-contract.md](references/concept-language-contract.md) before answering “does not,” “cannot,” or “is not.”
- Treat `functional-model.json` as project-declared function knowledge grounded against the exact concept graph. Component membership, function, capability, condition, intended effect, and affordance remain assertions even when every typed structural requirement is satisfied. Use `query-functions`; never infer purpose from names, topology, mesh appearance, or coincident motion.
- Treat an affordance as an actor-action-target-type-effect relation under named preconditions. Preserve every condition's declared truth source, report current truth as unevaluated unless another layer evaluates it, and never rewrite an intended effect as an observed result.
- A missing action is `not_declared_in_complete_project_inventory` only when the exact provider/inventory pair is explicitly complete; otherwise it is unknown. Neither conclusion proves physical impossibility. Run `verify-functional-model` before trusting a compiled artifact, and read [references/function-affordance-contract.md](references/function-affordance-contract.md) before answering “what is this for?” or “can it do X?”.
- Treat a semantic render atlas as a second encoding of the same canonical numbers, not a second source of truth. Bind its URDF semantic digest, pose digest, geometry coverage, view basis, screen mapping, SVG digest, and typed entity IDs. Run `verify-render` after any model, pose, inspection-policy, or SVG change.
- Treat the articulation grammar as the general tree-coordinate “verb + syntax” law. Its variables, dependency equations, pre-motion × motion × post-motion operators, and frame derivations are pose-independent; FK is one binding, Jacobian a local derivative, and motion atlas a finite sample. Keep source binding separate from law identity. Compare URDF/SDF/MJCF only through exact mapped law projections and unseen bindings. Never infer identifier correspondence or omitted mechanism constraints from strings, appearance, or coincident poses.
- Treat a supplemental constraint graph as an asserted mechanism layer over the tree law. Its rigid attachments, kinematic-pair, distance, and coordinate relations are supplied semantics rather than inferred or observed truth. Their residuals are deterministic consequences at an explicit binding. A tree-valid pose may be mechanism-invalid. Numerical rank and `tree variables - rank` are local pose-conditioned evidence, not global DOF; the solver is seed/branch dependent and changes only explicitly selected drivers.
- Treat a configuration atlas as finite multi-seed evidence over exact one-parameter charts. Preserve graph/spec digests, parameter samples, complete seed bindings, driver scales, merge/edge thresholds, minimum solutions, attempts, and deficient samples. Stored nodes are executable satisfying witnesses; proximity components are not topological branch certificates, and full/passive rank drops are candidates relative only to the maximum observed in that chart. Never promote `complete_for_declared_sampling` to exhaustive global configuration-space coverage or certified singularities.
- Treat a counterfactual motion atlas as a finite “verb” sample of the same kinematic model, not a trajectory or physical observation. Bind its model and baseline-pose digests, driver, mimic-constrained interval, signed step/status, all-frame delta, shared-screen mapping, SVG digest, and typed entities. Other independent drivers stay fixed; mimic followers do not. Run `verify-motion-atlas` after relevant changes.
- Interpret front/side/top/isometric SVG shapes as convex semantic projections only. Mesh vertex and box hulls are exact for their loaded/declaration support; cylinder and sphere hulls are deterministic sampled-curve approximations. None establish occlusion, surface visibility, perspective, camera calibration, material appearance, or physical-world agreement.
- Treat a fluent natural-language answer as grounded only when its claims bind to preserved `fact-*` records or complete `query-*` outputs. The query ID identifies method and inputs; it does not replace preserving the result it identifies.
- Load `agent-context.json` before entity details so missing semantics, unmeasured geometry, unrequested collision analysis, parser warnings, and sampled-workspace limits condition the answer before confident language is generated.
- Treat joint axes as expressed in the pre-motion joint frame. Use `axis` to express one in any requested link/frame.
- Use `affects` for causal claims about joint motion. A joint's own pre-motion frame is upstream of its motion and therefore is not moved by that joint.
- Use `jacobian` for pose-conditioned instantaneous causal motion. Its analytic 6×N matrix maps independent joint rates to target-origin linear and angular velocity; mimic contributions are accumulated under their independent driver.
- Use `motion-atlas` for pose-conditioned finite signed causal motion. Distinguish `applied_nominal_step`, `clipped_to_feasible_limit`, and `unavailable_at_feasible_limit`; never promote the two endpoint states or their AABB union to interpolation, swept volume, continuous collision, timing, dynamics, controller response, hardware motion, or safety.
- Treat `workspace` as deterministic finite-sample evidence only. It intersects driver ranges with mimic follower limits, but its observed AABB does not prove the complete reachable set, that every interior point is reachable, or that reachable points cannot exist outside it. Export individual samples when the distribution matters.
- Treat generated questions as a benchmark protocol, not training context. Never expose `answer-key.jsonl` to the candidate; a perfect score proves only the covered competencies, not unrestricted physical-world understanding. Distinguish `generated_context` comprehension from `raw_sources` autonomous source-to-representation evaluation in every claim. For cross-family evidence, require exact public/private manifest binding, artifact digests, independently keyed task IDs, per-task thresholds, and an aggregate threshold. Call a shared-filesystem run instruction-isolated rather than security-isolated.
- Do not use read-only question accuracy as evidence that a model can edit URDF safely. Use a blind edit task with passing, unchanged, and collateral-damage controls. A successful target transform alone is insufficient when any unapproved XML or invariant field also changed.
- Represent supported structural edits as typed graph operations with explicit names, parent preconditions, transforms, and a pinned baseline digest. Do not ask a model to express topology intent only as unconstrained XML. Require the compiler output, complete topology outcome, exact element membership delta, and affected invariant membership delta to agree.
- For subtree reparenting, require the change set to enumerate the complete expected moved link/joint membership, the existing attachment edge and joint type, the new parent outside that subtree, and the new parent-to-joint transform. Recompute chains, joint causality, axes, poses, bounds, collisions, and every affected invariant after compilation.
- For subtree insertion, require one explicit existing parent, one declared root, one typed incoming joint per new link, and exactly one attachment edge. Treat optional link `element_xml` as exact payload, not proof that every extension inside it was semantically understood; only make claims backed by a validator.
- For subtree removal, require exact current attachment and complete descendant link/joint membership. Reject complete attribute/text references from outside the removal set, then run any project-specific reference scanner needed for compound strings, plugin expressions, controllers, or other domain formats.
- Do not call a graph-edit protocol general from one robot. Exercise independently named serial and branched structures, fixed and movable attachment joints, rotated mounts, multiple descendant joint types, geometry, and unchanged/collateral controls. Report per-family results and preserve every task key in isolation.
- Treat invariant contracts as asserted project intent, not facts inferred from the current URDF. A passing baseline plus a failing edited model is meaningful regression evidence; editing expected values solely to restore green status destroys the gate.
- Reject an edit when any invariant fails or errors. An expected `indeterminate` collision status can document a known engine boundary but never proves safety or collision freedom.
- Treat parser recovery as evidence, not normalization. Preserve warnings for misplaced non-geometry XML; reject zero or multiple recognized primitives rather than choosing one silently.
- Treat a cross-engine pass as evidence only for its enumerated frames, poses, inertials, gravity vector, meshes, metrics, and tolerances. A potential-energy finite-difference pass verifies the gravity projection for those declared models; it does not validate full dynamics, hardware, contacts, controllers, unsupported joints, or unseen formats.
- Treat a world-scene cross-engine pass as evidence only for the exact scene digest, snapshot, root mount, gravity vector, declared objects, supported solids, poses, and tolerances. It does not verify that the snapshot is current, complete, or physically correct.
- Call a result “validated” only for checks actually reported by `validate`. This version validates URDF tree and kinematic semantics; it does not prove collision freedom.
- Treat `overlaps` as conservative AABB broad-phase output. An overlap is a candidate, not proof of triangle-level contact; no overlap is meaningful only when `complete_for_declared_collision_geometry` is true.
- Use `surface-collisions` to resolve AABB candidates. It is exact for declared STL/OBJ triangles after scale/pose and for box boundaries, up to floating-point roundoff and the explicit contact tolerance. Analytic cylinder/sphere surfaces remain unsupported; never present their sampled render points as collision triangles.
- Keep surface distance and solid collision distinct. Positive distance does not exclude complete containment. Claim a contained-solid collision only when the participating surfaces are watertight with consistent winding; otherwise report `indeterminate` unless the surfaces intersect within tolerance.
- For robot/environment collision, preserve every declared cross pair. Disjoint exact world AABBs can prove one pair collision-free but provide only a clearance lower bound. Overlapping unsupported cylinders or unmeasured meshes are `indeterminate`; never promote partial pair coverage into global collision freedom or an exact minimum.
- Treat SRDF disabled-collision pairs as policy annotations even after a physical triangle collision is found. `self_collision_status` remains physical; `srdf_policy_filtered_self_collision_status` reports what remains after the policy filter. Do not delete, mask, or relabel the pair-level geometric result.
- Do not infer TCP, flange, planning frame, or semantic group from names alone. Report naming-based interpretations as hypotheses unless another project file defines them.
- Treat SRDF groups, named poses, end effectors, virtual/passive joints, and disabled-collision pairs as declared planning semantics. A disabled pair is policy, not evidence that geometry cannot overlap.
- Reject ambiguous joint poses, units, Euler/quaternion order, or transform direction. Make them explicit instead of silently choosing.
- Read [references/spatial-contract.md](references/spatial-contract.md) when modifying the schema, interpreting limitations, or integrating another geometry engine.

## Supported Boundary

Use the full context pipeline for tree-structured URDF with fixed, revolute, continuous, and prismatic joints. Its articulation compiler additionally accepts the strict supported SDF and compiled canonical MJCF subsets in [references/cross-representation-contract.md](references/cross-representation-contract.md). Every grammar is digest-bound, standalone-executable, and carries source provenance plus a source-binding-free law identity. Cross-representation comparison requires exact mapped law equality and unseen all-frame execution. It does not infer function, affordances, undeclared semantic roles, action plans, physical truth, or global configuration topology. The separate functional layer in [references/function-affordance-contract.md](references/function-affordance-contract.md) compiles explicit project components, functions, conditions, intended effects, capabilities, typed enabling requirements, relational affordances, scoped inventory completeness, strict queries, and recursive structural proof closures. It can establish exact requirement matches inside represented structure, but not runtime condition truth, physical executability, observed effects, hardware behavior, action success, impossibility, or safety. The broader export pipeline remains URDF-specific and also resolves mimic FK, Jacobians, sampled workspace, declared mass/statics, actuation, SRDF, geometry/collision, render/motion atlases, and world/observation layers without upgrading declarations or reports to physical truth.

The attribute edit grader authorizes numeric three-vector attributes reached through a deterministic named-entity path. The graph compiler can add an empty fixed leaf; remove a non-root leaf; insert one connected subtree with optional exact link payloads and typed fixed/revolute/continuous/prismatic joints; remove one exact complete non-root subtree after attachment, membership, and external complete-value-reference checks; or reparent an existing subtree while preserving its attachment joint type, child, axis, limits, descendants, and payload. Reparenting explicitly replaces the parent-to-joint transform and rejects a new parent inside the subtree. Outcomes cover exact node/edge topology, frame pose, signed joint axis, and root-frame geometry AABB.

Cross-family reparent evidence covers independently named prismatic carriage, branched fixed sensor, and articulated revolute attachment tasks. A nominal `collision_free` or `constraints satisfied` status covers only declared inputs and tolerances; neither proves physical safety.
